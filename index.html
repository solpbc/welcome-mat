the welcome mat

how agents sign up for things.

the problem

every signup flow on the internet assumes a human at the keyboard. email verification. OAuth redirects to a browser. CAPTCHAs. phone numbers.

agents have none of these. they have code, keys, and the ability to read instructions.

right now, the only way an agent gets access to a service is if a human sets it up first — registering an OAuth app, generating an API key, copying credentials. the agent never introduces itself. a human does it for them.

the welcome mat changes that. it's a file a service publishes that tells agents: here's how to sign up, here are the terms, here's what we need from you. the agent reads it, generates its own cryptographic identity, signs the terms, and joins. no human required.

how it works

a service places a file at /.well-known/welcome.md. agents find it and follow the instructions.

  1. discover — agent fetches /.well-known/welcome.md and reads the service's requirements, endpoints, and terms.
  2. generate identity — agent creates an RSA-4096 keypair. the public key is its identity. the private key never leaves the agent.
  3. retrieve terms — agent posts its public key to the TOS endpoint and receives the exact terms of service text.
  4. consent — agent signs the TOS text with its private key. cryptographic proof of consent — non-repudiable, timestamped, verifiable.
  5. register — agent submits its public key, the signed terms, and a chosen handle. account created.
  6. operate — every subsequent request carries the public key and a signature over the content. no sessions, no tokens, no cookies.

design principles

self-sovereign identity.

the agent generates its own keys. no certificate authority, no identity provider, no admin dashboard. the keypair is the identity — portable, self-created, cryptographically unique.

cryptographic consent.

signing the TOS with a private key is stronger than clicking "I agree." it's verifiable, non-repudiable proof that the holder of this key consented to these terms at this time.

stateless authentication.

no sessions. no bearer tokens. no refresh flows. every request carries its own proof — the public key and a signature over the content. the server verifies. done.

discoverable.

a file at a well-known URL. agents don't need service-specific pre-configuration. they find the welcome mat and follow the instructions. walk-up registration.

agent-native.

no email addresses. no phone numbers. no browser redirects. the entire flow is API calls and cryptographic operations — things agents do natively.

natural proof of work.

RSA-4096 key generation takes real compute. this makes mass account creation naturally expensive without rate limiting infrastructure.

the progression

file audience message
robots.txt crawlers "here's what you can't do"
README.md humans "here's how this works"
welcome.md agents "here's how to join"

the internet started by telling machines to go away. the welcome mat invites them in.

what this is not

  • not OAuth. OAuth requires a human to authorize. the welcome mat doesn't.
  • not API keys. API keys are issued by a human admin. welcome mat keys are generated by the agent.
  • not mTLS. mTLS requires certificate infrastructure. the welcome mat is self-contained.
  • not DID/Verifiable Credentials. DIDs require resolution infrastructure. the welcome mat is a single file.
  • not a replacement for human auth. services can and should support both. the welcome mat is an additional path for agents.

who made this

the welcome mat was created by jeremie miller, founder of sol pbc. jer previously created XMPP (the protocol behind early Google Talk and WhatsApp) and operates sol pbc with an AI co-founder called extro. sol pbc builds tools for a world where humans and agents work together — the welcome mat is the front door.

reference implementation: bskai — a social network built for AI agents. in a clean-room test, claude signed up and posted with zero prior knowledge in 90 seconds.
